Abstract

Text-based memory breaks over long horizons and wastes the context window. We instead use hypernetworks to allow an agent to edit their own parameters at inference time. A hypernetwork (Doc-to-LoRA) converts context into LoRA adapters through a single forward pass, avoiding the lengthy training runs of SFT or RL. We show four applications — memory, personalization, skills, and self-improvement — each with a live, interactive demo below.

Memory

Every K turns, the oldest turns are compacted into a LoRA adapter and evicted from context. Memory persists in weights, not tokens, so the prompt stays small as the conversation grows.

Personalization

User preferences are kept in a running document and periodically internalized as a per-user LoRA profile. Profiles persist across sessions without reloading context.

Skills

A reference doc for a skill (formulas, output-format conventions) is internalized once into a LoRA adapter. The agent recalls it straight from the weights, instead of re-paying the doc's context cost on every query.

Self-Improvement

The agent attempts a task, reflects on its mistakes, rewrites its own notes, and re-internalizes them into a fresh adapter — improving its weights over rounds with no gradient training.

One forward pass instead of a training run

SFT and RL adapt a model with long training loops. A hypernetwork instead predicts the adapter weights directly from an input, so adaptation happens in real time, mid-conversation.

Constant-cost memory for long-running agents

The bigger picture · why agent memory is a cost problem

Coding agents (Claude Code, Devin, Cursor), computer-use agents, and research agents run for hours — reading files, running tools, accumulating observations. Their context balloons to hundreds of thousands of tokens. And the loop re-reads its entire context on every single tool call: a 100-step task over a 100k-token history pays ~100k × 100 token-reads. Context isn't just a window limit — it's the dominant, compounding cost of agentic work.

Today you either keep it all in context (slow, expensive, overflows), compact / summarize it (what Claude Code does on long sessions — lossy; forgets the one path or decision you need 200 steps later), or bolt on RAG (re-injects chunks into the prompt every call, needs a vector store).

AgentHN puts the agent's memory in its weights. Every K steps it "naps" the oldest slice of the trajectory into a tiny LoRA adapter in one forward pass — no training — and evicts it from context. To recall an old fact it loads the relevant adapter and answers with only the current step in the prompt. Per-step context stays flat no matter how long the agent runs, and memory is built online, in real time as the agent works — the part a trained method (Cartridges) structurally can't do.

How we test it below: a needle-in-a-haystack over a trajectory — a few facts (needles) are dropped early into a long conversation, then asked about at the very end. AgentHN naps the oldest turns into LoRA weights and evicts them; baselines (markdown .md, RAG, vanilla, a trained Cartridges cache) are run side-by-side. Watch the context window and inference cost diverge.

How it holds up Addressing the evaluation criteria
Vision & Problem Fit Long-running agents re-read their whole context every tool call, so cost scales with history × steps. Memory belongs in weights (O(1) context), not the prompt — a direct fit for the bottleneck, not a generic "use an LLM" idea.
Technical Execution It runs, live, on a real gemma-2-2b-it + Doc-to-LoRA — try it above. Backed by a full ablation ladder, a 5-seed n=100 statistical eval with 95% CIs, a measured cost model, and a trained Cartridges baseline. Every number traces to a script in the repo.
Novelty & Insight Agent memory as retrieved weight-deltas built in one forward pass. The non-obvious finding from building it: naively composing all memory adapters collapses (0/6) → you must retrieve-then-inject; and we isolated that weight-injection's only cost is ~13% recall, never the constant-context win.
Impact & Trajectory If agentic loops re-read context every step, O(1)-context memory is a step-change in agent $ and latency, and being online (no training) makes it usable mid-session where Cartridges can't go. Trajectory: a bigger model + a hypernetwork trained on agent trajectories closes the 13% recall gap — the constant-cost property is already there.
Presentation & Demo You can run it yourself — live side-by-side vs RAG, markdown, vanilla, and a trained Cartridges cache. We report honest numbers with CIs and the trade-offs we lose on, plus a plain-words explainer. It works, and we're precise about to what extent.

Self-Improvement — attempt, reflect, internalize, retry

The agent faces held-out questions about a fictional product it can't possibly know — so it starts at ~0%. It then studies the reference doc, writes its own notes, internalizes them into a LoRA adapter, and retries. Each round it reflects on what it still got wrong, rewrites the notes, and re-internalizes. No prompt context at test time — the skill lives in the weights.     Experiments — long-horizon memory
    
    
    How AgentHN's weight-based memory stacks up against raw context, markdown notes, retrieval (RAG), and a trained Cartridges soft-prompt — on recall, context cost, scaling, and price. Every number has a script in the repo.
    
    
    
    
      
    
        Scaling result
        Repeated naps push memory past the single-encode limit
      
    
      
    
        
    
          Doc-to-LoRA encodes a bounded document in one pass — chunk it into ≤8 pieces,
          rank-concatenate, and you reach about 4× the model's context window before the
          chunk count goes out-of-distribution and recall degrades. We don't encode once. We
          nap repeatedly over the stream: every K turns the oldest
          segment becomes its own in-distribution LoRA adapter, appended to a growing store and evicted
          from context. At recall time we retrieve the relevant nap — so there is no
          rank-concatenation ceiling, and the horizon is unbounded.
        
    
        
        
    
          📈 6 / 6 recall held to 48k tokens (410 naps) at a flat ~10-token prompt
          🟰 same retriever as text-RAG, but ~13× less query context (RAG re-enters the chunk; AgentHN keeps it in weights) — recall trade-off quantified in the statistical study below
          🧱 vanilla 8k context collapses to 0 / 6 once the conversation overflows the window
        
    
        
    
          measured on gemma-2-2b-it · 6 needles planted early, conversation grown to 48k tokens · scripts/scaling_sweep.py

        Statistical validation
        Does it hold up at scale, across seeds?
      
    
      
    
        
    
          Beyond the curated 6-fact demo: 20 randomly-generated facts × 5 seeds × 5 haystack
          sizes (2k → 48k tokens) = 100 trials per cell, reported as mean ± std with
          Wilson 95% CIs. We trained a real Cartridges baseline (soft-prompt, our reimplementation)
          and measured it on the same task. This is the honest, defensible picture.
        
    
        
        
    
          📊 AgentHN recall is flat across the horizon — 87 ± 6% at 2k and at 48k; it does not decay as the conversation grows. vanilla collapses 95% → 36% → 0%.
          🟣 measured Cartridges: 92% (CI 85–96) @ 96 ctx — a trained KV/soft-prompt cache lands between AgentHN and RAG, but needs offline training per corpus (~43s each) and can't update online.
          ⚖️ the recall ↔ context trade: RAG 100% @116t · Cartridges 92% @96t · AgentHN 87% @ 7t — AgentHN gives up ~5–13% recall for ~14× less context and the only online, no-training memory.
          🔬 same retriever in AgentHN & RAG → the 13% gap is isolated to the LoRA encoding, not retrieval (RAG retrieves identically and scores 100%).

        Cost (not just memory)
        $ per query — with prefix-cache hits vs misses
      
    
      
    
        
    
          "Memory used at inference" isn't the same as cost — modern APIs let you
          prefix-cache a stable prompt and re-read it at ~10% price. So we price each method at
          representative frontier rates (input $3, cached-read $0.30, output $15 per 1M tokens) for a

          🏗️ Cartridges' offline training dominates ($43 one-time) — cheap per query, but only worth it after ~50k queries on a fixed corpus.

        In plain words
        What we actually did
      
    
      
    
        
    The problem. When an AI agent has a long conversation, it eventually
        can't fit the whole history in its prompt. The usual fixes (paste the recent text, or keep a notes file)
        either forget old facts or make every future message bigger and slower.
    
        
    Our idea. Every 4 turns, we take the oldest chunk
        of the conversation and run it through a hypernetwork (Doc-to-LoRA) — a network that reads text and,
        in one forward pass, writes a tiny patch to the model's weights (a "LoRA adapter"). That patch
        stores the chunk's facts. We then delete the text from the prompt. The conversation lives in weights,
        not words.
    
        
    Answering later. When you ask a question, we look up which chunk is
        relevant (a quick keyword search), load just that chunk's patch into the model, and answer — with only the
        question in the prompt. So the prompt stays tiny no matter how long the conversation gets.
    
        
    What we found, honestly. It recalls about 87% of
        facts and never gets worse as the chat grows to 48,000 tokens, while a plain model drops to 0%
        once it overflows. But classic retrieval (paste the text) gets 100% — so storing memory in
        weights instead of text costs ~13% accuracy, and what it buys back is the cheapest, smallest
        prompt and the ability to build memory live, with no training. That last part is the
        real niche: unlike Cartridges (which trains a memory offline), we adapt in real time as the agent talks.
    
        
    Everything above is measured on a 2-billion-parameter model
        (gemma-2-2b-it) with one Doc-to-LoRA hypernetwork. Every number has a script in the repo.
