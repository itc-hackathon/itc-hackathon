Abstract

Text-based memory breaks over long horizons and wastes the context window. We instead use hypernetworks to allow an agent to edit their own parameters at inference time. A hypernetwork (Doc-to-LoRA) converts context into LoRA adapters through a single forward pass, avoiding the lengthy training runs of SFT or RL. We show three applications across memory, personalization, and skills, each with a live, interactive demo below.

Long-horizon memory

Every K steps, the oldest turns are compacted into an adapter and evicted from context. Memory persists in weights, not tokens, mitigating context window overflow.

Permanent personalization

During conversations, user preferences are updated in a document and periodically internalized as per-user LoRA profiles. Profiles persist across sessions without needing to reload context.

Self-improving skills

A reference doc for a skill (formulas, output-format conventions) is internalized once into a LoRA adapter. The agent recalls it straight from the weights afterward, instead of re-paying the doc's context cost on every query.

One forward pass instead of a training run

SFT and RL adapt a model with long training loops. A hypernetwork instead predicts the adapter weights directly from an input, so adaptation happens in real time, mid-conversation.

Scaling result Repeated naps push memory past the single-encode limit

Doc-to-LoRA encodes a bounded document in one pass — chunk it into ≤8 pieces, rank-concatenate, and you reach about 4× the model's context window before the chunk count goes out-of-distribution and recall degrades. We don't encode once. We nap repeatedly over the stream: every K turns the oldest segment becomes its own in-distribution LoRA adapter, appended to a growing store and evicted from context. At recall time we retrieve the relevant nap — so there is no rank-concatenation ceiling, and the horizon is unbounded.

📈 6 / 6 recall held to 48k tokens (410 naps) at a flat ~10-token prompt 🟰 same retriever as text-RAG, but ~13× less query context (RAG re-enters the chunk; NapLoRA keeps it in weights) — recall trade-off quantified in the statistical study below 🧱 vanilla 8k context collapses to 0 / 6 once the conversation overflows the window

Where NapLoRA sits vs RAG and Cartridges — four ways to give a model memory

The honest framing isn't "we beat everyone" — it's which axis you optimize. Cartridges (HazyResearch) trains a small KV cache offline per corpus and matches in-context quality; RAG pastes the retrieved text; NapLoRA predicts a LoRA in one forward pass. They trade off creation cost, whether they can run online, recall, and per-query cost differently.

The takeaway: Cartridges wins on quality but needs expensive offline training and can't update mid-conversation. NapLoRA is the online / real-time point — it builds memory in a single forward pass as the agent talks, at the lowest per-query cost — and pays for that amortization with ~13% recall. Different tools for streaming agent memory (NapLoRA) vs a fixed corpus queried millions of times (Cartridges).

Cost (not just memory) $ per query — with prefix-cache hits vs misses

"Memory used at inference" isn't the same as cost — modern APIs let you prefix-cache a stable prompt and re-read it at ~10% price. So we price each method at representative frontier rates (input $3, cached-read $0.30, output $15 per 1M tokens) for a 48k-token history queried 1,000 times.

💾 prefix caching helps vanilla 10× (144µ$ → 14.9µ$ / query) — but it still re-reads the 48k prefix every query. ⚡ NapLoRA is cheapest per query (0.47µ$) — it only processes the question; the memory is a weight swap, not tokens. 🏗️ Cartridges' offline training dominates ($43 one-time) — cheap per query, but only worth it after ~50k queries on a fixed corpus.

In plain words What we actually did

The problem. When an AI agent has a long conversation, it eventually can't fit the whole history in its prompt. The usual fixes (paste the recent text, or keep a notes file) either forget old facts or make every future message bigger and slower.

Our idea. Every 4 turns, we take the oldest chunk of the conversation and run it through a hypernetwork (Doc-to-LoRA) — a network that reads text and, in one forward pass, writes a tiny patch to the model's weights (a "LoRA adapter"). That patch stores the chunk's facts. We then delete the text from the prompt. The conversation lives in weights, not words.

Answering later. When you ask a question, we look up which chunk is relevant (a quick keyword search), load just that chunk's patch into the model, and answer — with only the question in the prompt. So the prompt stays tiny no matter how long the conversation gets.

What we found, honestly. It recalls about 87% of facts and never gets worse as the chat grows to 48,000 tokens, while a plain model drops to 0% once it overflows. But classic retrieval (paste the text) gets 100% — so storing memory in weights instead of text costs ~13% accuracy, and what it buys back is the cheapest, smallest prompt and the ability to build memory live, with no training. That last part is the real niche: unlike Cartridges (which trains a memory offline), we adapt in real time as the agent talks.

Everything above is measured on a 2-billion-parameter model (gemma-2-2b-it) with one Doc-to-LoRA hypernetwork. Every number has a script in the repo.

Self-improving skills — attempt, reflect, internalize, retry live

The agent faces held-out questions about a fictional product it can't possibly know — so it starts at ~0%. It then studies the reference doc, writes its own notes, internalizes them into a LoRA adapter, and retries. Each round it reflects on what it still got wrong, rewrites the notes, and re-internalizes. No prompt context at test time — the skill lives in the weights. The accuracy curve is the real run, dips and all.
